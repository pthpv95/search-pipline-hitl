Phase 1 — Multi-agent research pipeline

Foundation
Week 1
Project skeleton + state design
Define graph, state schema, agent roles
▾
Day 1–2
Define the LangGraph state schema. What does the graph need to know at every step? Topic, search results, synthesis draft, follow-up questions, final report. Write this as a TypedDict — not code yet, just the shape.
Day 3–4
Scaffold the graph with 3 empty nodes: search_agent, synthesis_agent, report_agent. Wire edges between them. Add a human_review interrupt after synthesis. Run it end-to-end with stub functions — the flow should work before any real logic exists.
Day 5
Write a test that runs the full graph with a fixed topic. Assert each node receives the right state slice. You want a safety net before adding real LLM calls.
◎Deliverable: runnable graph skeleton, typed state, one passing integration test
Week 2
Search agent — tools + structured output
Web search, result parsing, gap detection
▾
Day 1–2
Implement the search agent with Claude tool-use. Give it 2 tools: web_search and fetch_page. The agent decides how many searches to run. Use model_router to let it call tools in a loop until it has enough context.
Day 3
Force structured output: the agent must return a Pydantic model with findings: list[Finding], gaps: list[str], sources: list[Source]. Use with_structured_output(). No freeform text leaving this node.
This is where most people skip validation — don't.
Day 4–5
Add retry logic. If structured output fails validation, retry once with a corrective prompt. Log failures. This is your first taste of reliability engineering — the skill that separates production agents from demos.
◎Deliverable: search agent that returns validated structured output with retry on failure
Core logic
Week 3
Synthesis agent + follow-up loop
Multi-turn reasoning, gap filling
▾
Day 1–2
Build the synthesis agent. It takes search results and produces a draft with explicit knowledge gaps. It should also generate 2–3 follow-up search queries for the gaps it found.
Day 3
Add a conditional edge: if gaps exist and loop count is under 2, route back to the search agent with the follow-up queries. This is the agentic loop — the hardest concept to get right. Control the loop count in state, not in the agent.
Day 4–5
Test with 3 different topics. Watch how the loop behaves. Does it over-search? Under-search? Tune the gap detection prompt. The goal: 1–2 loop iterations for most topics, not 0 and not infinite.
◎Deliverable: working synthesis loop with controlled iteration, conditional routing
Week 4
Human-in-the-loop checkpoint
Interrupt, review, resume pattern
▾
Day 1–2
Implement the HITL checkpoint after synthesis using LangGraph's interrupt(). Show the human the draft + gaps. They can approve, edit the draft, or add more search queries. This is the most impressive feature to demo — it shows you understand agentic vs automated.
Day 3
Persist state with MemorySaver so the graph can be paused and resumed across Python process restarts. You should be able to kill the process, restart, and resume from the checkpoint. This is table stakes for production agents.
Day 4–5
Build a dead-simple CLI interface for the HITL step. Not a UI — just a prompt that shows the draft and takes input. Focus on the pattern, not the presentation.
◎Deliverable: pausable graph with persistent checkpoints, working HITL review step
Polish + depth
Week 5
Report agent + output quality
Final report, citations, formatting
▾
Day 1–2
Build the report agent. It takes the approved synthesis and produces a clean markdown report with sections, inline citations, and a source list. The report should be usable by a non-technical person.
Day 3
Add report templates: "executive brief" (1 page, key points only) vs "deep dive" (full report with all sources). The user picks at graph initialization. This teaches you parameterized graph behavior.
Day 4–5
Run the full pipeline on 5 real topics. Save inputs and outputs. These become your eval dataset next week. Don't skip this — you'll need real examples to measure improvement.
◎Deliverable: polished report output in 2 formats, 5 real end-to-end runs saved
Week 6
Evals, cost tracking, final polish
Measure quality, optimize, make it showable
▾
Day 1–2
Write a simple eval script using your 5 saved runs. For each run, score: (1) source quality (are citations real?), (2) coverage (did it address the topic?), (3) loop efficiency (how many iterations?). Simple heuristics are fine — the habit of measuring is what matters.
Day 3
Add token tracking per node. You should know exactly what each step costs and how long it takes. Log this to a simple JSON file per run. Two questions to answer: which node is the most expensive? Which is the slowest?
Day 4–5
Write a short README covering: what it does, how to run it, one real example with actual output. Record a 2-minute screen demo walking through a run. This is your portfolio artifact. If someone asks what you've built with LangGraph, you send this link.
◎Deliverable: eval script, cost tracking, documented repo, screen demo recorded
